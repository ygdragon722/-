后续若扩写剧情，继续沿用这一套事件元数据。

1. 扩展类型定义：
   - `GameEvent.id`
   - `GameEvent.title`
   - `GameEvent.stage`
   - `GameEvent.hint`
   - `EventChoice.tone`
   - `GameState.completedEvents`
   - `GameState.mbtiInsight`

2. 给现有 `EVENT_DB` 补事件元数据：

3. 修改事件触发逻辑：
   - 探索角色地点时，优先找“未完成且条件已达到”的下一事件。
   - 如果下一事件条件未满足，允许显示普通闲聊或提示，不要重复刷已完成事件奖励。
   - 完成选择后把事件 id 写入 `completedEvents`。

4. 兼容旧存档：
   - `loadSave` 时如果没有新字段，补默认值。
   - 旧存档不应导致白屏或类型错误。

## P2：AVG 对话演出

目标：把现在偏弹窗式的 `EventModal` 改成更像 90 年代中文 AVG 的对白层，让剧情发生时仍然是满屏画面。

1. 对话层结构：
   - 背景继续使用当前地点图。
   - 左右显示相关人物立绘。
   - 底部显示对白框、角色名、事件标题。
   - 选项出现在对白框上方或右侧，不再使用居中大弹窗。

2. 事件表现：
   - 普通事件只显示对白层。
   - 重要节点可切换事件 CG。
   - 结算结果显示为简短浮层或日志，不打断画面。

3. 交互：
   - 点击地点触发剧情后进入对白演出。
   - 玩家选择后播放反馈，再回到大观园主界面。
   - 保留现有事件数据结构，不在第一版重写剧情引擎。

   - 第一版只做数值与日志反馈，不急着做复杂动画。

## P5：内容扩写

1. 每位角色补到 5 个剧情节点：
   - 初遇
   - 试探
   - 冲突
   - 名场面
   - 结局前夜

2. 优先扩写顺序：
   - 黛玉、宝钗、晴雯：主线辨识度最高。
   - 探春、湘云：机制差异明显。
   - 袭人、妙玉：补足节奏和结局层次。

3. 文案原则：
   - UI 和选项可现代。
   - 主线对白稍微古典。
   - MBTI 用来解释关系互动，不要把角色写成标签说明书。
