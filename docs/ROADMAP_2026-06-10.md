## 项目定位

《红楼幻梦》的目标是学习并复刻智冠科技 1997 年《红楼梦》那类恋爱养成游戏的核心体验：玩家扮演贾宝玉，在大观园中通过日程、探索、任务和对话推进剧情，攻略金陵十二钗相关角色。

本项目的创新点是把人物 MBTI 与现代情感语境融入红楼人物关系。现代词可以存在于 UI 标签、系统提示和轻喜剧选项里；主线对白仍应尽量保留红楼的细腻和古典气质，避免人物被段子化标签吞掉。

0.3.0 优先桌面网页版体验：跑得通、看得清、目标明确、界面友好。PWA 和手机版宣传降级，现有 PWA 配置暂不急删，除非它开始妨碍桌面体验或维护效率。

## 当前判断

项目已有核心骨架：30 天养成、七位女主、MBTI 标签、地点探索、随机事件、大事件、多结局、立绘、场景和事件 CG。

当前短板不是图片资产，也不是能否运行，而是玩家目标感不足。好感阈值、道具需求、才学门槛主要藏在事件里，玩家不容易知道下一步该追谁、买什么、补什么属性。

因此 0.3.0 的核心不是继续堆剧情文本，而是先补上“剧情进度系统 + 梦册任务 UI + MBTI 偏好反馈”。否则新增剧情越多，越容易被当前“只按好感阈值取事件”的逻辑卡住。

## 0.3.0 核心目标

把游戏从“数值事件 demo”升级为“有明确路线、任务簿和章节节奏的桌面网页版恋爱养成游戏”。

验收标准：

1. 玩家能在界面上看到每位角色的路线进度。
2. 玩家能知道下一段剧情大致差什么条件，而不是点进事件才发现条件不足。
3. 已完成剧情不会重复刷奖励。
4. MBTI 不只是人物卡标签，而会影响选项反馈、理解度或额外好感。
5. 第 15 天和第 25 天成为明确的结构锚点，而不是孤立弹窗。

## 路线与任务簿

任务簿命名建议为“梦册”。第一版不做独立大页面，先做桌面右侧常驻面板或主界面内的固定任务区。

展示策略：

- 每位角色显示 5 个剧情节点：初遇、试探、冲突、名场面、结局前夜。
- 已完成节点明确打勾。
- 未完成节点灰色显示，保留标题或模糊标题。
- 条件采用“部分提示”：例如“需提升才学”“需备一件清雅礼物”“好感尚浅”，不必全部精确剧透。
- 当前可触发节点高亮。
- 地图卡同步显示“下一剧情可触发/还差什么”。

路线策略采用“软专一”：

- 第 1-15 天允许玩家广泛认识多位角色。
- 第 15 天海棠诗社后，系统根据好感、MBTI 理解度和关键选择推荐一个主路线。
- 玩家可以继续支线互动，但深层结局前夜更偏向主路线，鼓励多周目。

## MBTI 机制

0.3.0 先做轻量机制，不新增过重的人格系统。

为选项增加 `tone`，角色有偏好 tone。选中匹配 tone 时，给予更好对白反馈、额外好感或“理解度”提升；选中冲突 tone 时，好感较少、无理解度，严重时触发负反馈。

建议 tone：

| tone | 含义 | 典型角色 |
|---|---|---|
| `empathy` | 情绪共鸣、精神理解 | 黛玉 |
| `planning` | 上进可靠、现实规划 | 宝钗 |
| `play` | 即时快乐、松弛陪伴 | 湘云 |
| `reform` | 事业资源、改革同盟 | 探春 |
| `care` | 稳定照顾、日常信任 | 袭人 |
| `rebel` | 真诚纵容、叛逆同频 | 晴雯 |
| `aesthetic` | 审美理解、精神洁癖 | 妙玉 |

角色偏好表：

| 角色 | 偏好维度 | 正向行为 | 低效/负向行为 |
|---|---|---|---|
| 黛玉 | 情绪共鸣、精神理解 | 共情、读诗、陪伴哀愁 | 讲道理、粗暴开解 |
| 宝钗 | 上进可靠、现实规划 | 读书上进、讨论家业 | 只谈风月、摆烂摸鱼 |
| 湘云 | 即时快乐、松弛陪伴 | 饮酒、游玩、轻松同行 | 说教、过度算计 |
| 探春 | 事业资源、改革同盟 | 支持改革、出钱出力 | 和稀泥、反对变革 |
| 袭人 | 稳定照顾、日常信任 | 接受关心、日常守信 | 忽冷忽热、拒绝照顾 |
| 晴雯 | 真诚纵容、叛逆同频 | 替她说话、不拿规矩压人 | 站王夫人规训她 |
| 妙玉 | 精神洁癖、审美理解 | 品茶、赏梅、谈诗词 | 谈俗务、表现粗鄙 |

## 章节节奏

建议保持 30 天结构，但让章节感更明显。

| 日期 | 阶段 | 目标 |
|---|---|---|
| 第 1-7 天 | 熟悉园子 | 认识角色，学习基础资源循环 |
| 第 8-14 天 | 确定路线 | 玩家开始形成偏好，梦册提示路线方向 |
| 第 15 天 | 海棠诗社 | 中点考核，根据才学和角色理解度给奖励 |
| 第 16-24 天 | 情感深化 | 进入冲突与名场面，铺垫抄检危机 |
| 第 25 天 | 抄检大观园 | 全局危机，检验资源、才学和关系 |
| 第 26-30 天 | 结局选择 | 结局前夜，收束主路线 |

海棠诗社建议：

- 考核才学，也考角色偏好理解。
- 通过后解锁高级剧情提示或关键道具。
- 失败不应卡死，但会提高后续路线难度或错过部分奖励。

抄检大观园建议：

- 作为全局主线危机保留在第 25 天。
- 第 16-24 天通过固定日期事件和环境变化铺垫：王夫人疑心、凤姐账目、晴雯流言、宝钗劝告、探春改革。
- 铺垫事件采用“固定日期 + 路线相关”的形式，例如第 18 天必触发晴雯流言，但内容根据晴雯/袭人/黛玉关系变化。

## 技术设计草案

优先补数据结构，再补界面。

建议新增或扩展：

```ts
type HeroineId = 'daiyu' | 'baochai' | 'xiangyun' | 'tanchun' | 'xiren' | 'qingwen' | 'miaoyu';
type ChoiceTone = 'empathy' | 'planning' | 'play' | 'reform' | 'care' | 'rebel' | 'aesthetic';

interface GameEvent {
  id: string;
  title: string;
  stage: 1 | 2 | 3 | 4 | 5;
  req: number;
  hint?: string;
  image?: string;
  choices: EventChoice[];
}

interface EventChoice {
  tone?: ChoiceTone;
  text: string;
  req?: ChoiceReq;
  cost?: ChoiceCost;
  reward?: ChoiceReward;
  reply?: string;
}

interface GameState {
  completedEvents: string[];
  routeFocus?: HeroineId;
  mbtiInsight: Record<HeroineId, number>;
  storyFlags: Record<string, boolean>;
}
```

实现原则：

- `completedEvents` 先解决剧情重复刷奖励问题。
- 梦册可以从 `EVENT_DB` 派生，不必一开始另建复杂任务系统。
- `routeFocus` 可以后置到海棠诗社之后再启用。
- 旧存档读取时要兼容缺失字段，给默认值。
